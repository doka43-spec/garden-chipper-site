## 🚨 ГЛАВНОЕ И НЕНАРУШАЕМОЕ УСЛОВИЕ ПРОЕКТА 🚨

# САЙТ ДОЛЖЕН БЫТЬ ПОЛНОСТЬЮ АВТОНОМЕН И НЕЗАВИСИМ ОТ ПЛАТФОРМЫ POEHALI.

**Это условие №1. Оно важнее любого другого требования.**
Платформа Poehali используется ТОЛЬКО как среда разработки и сборки.
В рабочем (опубликованном) сайте на Бегете НЕ ДОЛЖНО БЫТЬ НИ ОДНОГО следа платформы.

### ❌ КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО:
- Любые обращения к доменам poehali.dev, *.poehali.dev, functions.poehali.dev
- Использование CDN платформы (cdn.poehali.dev) для картинок, шрифтов, файлов
- Backend-функции платформы (Cloud Functions) — даже одна функция ломает автономность
- Хранение секретов/API-ключей платформы в работающем сайте
- Любые fetch/axios запросы на инфраструктуру Poehali
- Файлы func2url.json, ссылки на /api платформы в production-билде
- Зависимость от БД платформы (PostgreSQL Poehali)

### ✅ РАЗРЕШЕНО ТОЛЬКО:
- **Хостинг:** Бегет (beget.com), папка `public_html`
- **Домен:** rubitel.ru (собственный, привязан напрямую к Бегету)
- **Картинки:** GitHub + jsDelivr CDN (cdn.jsdelivr.net)
- **Письма:** mail.php на Бегете
- **SEO:** Яндекс.Вебмастер привязан к собственному домену

### 🎯 Принцип проверки:
Если завтра платформа Poehali исчезнет — сайт rubitel.ru должен продолжать работать без единого сбоя.
Это критерий правильности любого изменения.

## Картинки — GitHub + jsDelivr CDN
Репозиторий: https://github.com/doka43-spec/rubitel-images
CDN-ссылка: `https://cdn.jsdelivr.net/gh/doka43-spec/rubitel-images/имя_файла`

### Как добавить новую позицию в каталог (пошагово):

**Шаг 1 — Подготовить фото:**
1. Сжать фото на https://imagecompressor.com/ru/ — качество 80%
2. Дать понятные имена файлам: `название-модели-1.jpg`, `название-модели-2.jpg` и т.д.

**Шаг 2 — Загрузить фото на GitHub:**
1. Открыть https://github.com/doka43-spec/rubitel-images
2. Нажать **Add file → Upload files**
3. Перетащить файлы в окно браузера
4. Нажать зелёную кнопку **Commit changes**

**Шаг 3 — Добавить позицию в код (на платформе poehali.dev):**
1. Открыть файл `src/components/shared.tsx`
2. В массиве `PRODUCTS` добавить новый объект по образцу существующих
3. Для картинок использовать: `${GH}/имя_файла.jpg`

**Шаг 4 — Опубликовать:**
1. Сделать билд на платформе
2. Скачать билд → загрузить содержимое `dist/` в `public_html/` на Бегете

**Картинки на GitHub появляются сразу, билд на Бегете нужен для изменений в коде.**

### Важно:
- Папки `images/` на Бегете больше нет — все картинки только на GitHub
- jsDelivr кэширует файлы. Если заменил картинку с тем же именем — новая появится через ~24 часа
- Чтобы обновить картинку мгновенно — дай ей новое имя

## Правила при изменениях
- Письма: только через `mail.php` (не через backend-функции платформы)
- Никаких REVIEWS_API_URL и других внешних API

# 📜 ИСТОРИЯ ПРОЕКТА

## Этап 1 — Базовая разработка
- Создан SPA на Vite + React + TypeScript + Tailwind
- Стек: shadcn/ui, lucide-react, react-router-dom
- Тёмный индустриальный дизайн (угольный фон + жёлтые акценты)
- Основные секции: Hero, каталог рубителей, запчасти, отзывы, контакты

## Этап 2 — Переход на автономность
- Отказ от backend-функций платформы Poehali
- Картинки перенесены на GitHub (репозиторий `doka43-spec/rubitel-images`) + jsDelivr CDN
- Отправка писем — через `mail.php` на Бегете (заявки и отзывы → vyatkalux@yandex.ru)
- Хостинг переведён на Бегет (`public_html`), домен `rubitel.ru` привязан напрямую

## Этап 3 — SEO и индексация
- Загружен favicon.ico на Бегет
- Сайт добавлен в Яндекс.Вебмастер
- Главная страница отправлена на переобход
- Sitemap.xml загружен (5 страниц), статус OK
- Главное зеркало: `https://rubitel.ru` (HTTPS, без www)
- Schema.org: Organization, Product
- Мета-теги: title, description, OG, Twitter, canonical

## Этап 4 — Аудит проекта (текущий)
Проведён комплексный анализ. Выявлены приоритетные направления улучшений:
доступность (a11y), контрастность, локальный SEO, производительность, мобильный UX.
